# Gecko — Architecture Improvement Roadmap

*Lead architect synthesis. Every recommendation is grounded in code cited by file:line in the verified findings. Hard constraints (Engine Data Fidelity, exact engine-ID preservation on read/write, Qt parent-ownership as idiomatic, reuse of BaseWidget/BasePanel/ThemeManager/MimeTypes/EventBus/ObjectCommandController over parallel mechanisms) are honored throughout. Where a finding's original recommendation conflicted with a hard constraint or the verified code, the corrected guidance from the verification notes is used.*

---

## 1. Executive Summary — Highest-Leverage Themes

1. **Stop the active data-loss and crashes first.** Three confirmed bugs are shipping today: `ProWriter::writeWallData` zeroes a wall prototype's `flagsExt`/`SID` on every save (`ProWriter.cpp:399-400`) — a direct Engine-Data-Fidelity violation; `Loader::setStatus/setProgress` write shared `std::string`s with no lock while the UI thread reads them under a `shared_lock` (`Loader.h:38-44`) — a genuine data race + return-by-reference use-after-free; and `MapLoader::loadFromFilesystem` runs the FRM-preload loop outside any try/catch (`MapLoader.cpp:148`), so a single missing art asset escapes the worker thread and calls `std::terminate`. These are small, high-confidence, ship-now fixes.

2. **Break the EditorWidget ↔ managers ↔ MainWindow coupling cycle via the DI seam that already exists.** `ObjectCommandController` (`ObjectCommandController.h:37-44`, commit 2d5f7e7) already takes borrowed refs instead of an `EditorWidget*`. The remaining managers (`SelectionManager`, `DragDropManager`, `TilePlacementManager`, `ExitGridPlacementManager`) hold raw back-pointers into a ~73-method accessor surface. Migrating them to the established context-injection pattern is the single biggest structural unlock — it dissolves the god-hub, severs the compile-time `SelectionManager ↔ EditorWidget` cycle, and is the prerequisite for headless testing and the library split.

3. **Make config and notification injectable / honest.** `Settings::getInstance()` is reached at 17 sites with no injection seam (`Settings.cpp:35`); `EventBus` is a write-only singleton with 10 publishers and zero subscribers, duplicating Qt signals (`EventBus.h`). Mirror the existing `GameResources` shared_ptr DI for Settings; delete EventBus outright. Both are explicit PLAN.md item #2.

4. **Decompose the build into one-way layers so coupling is enforced, not just discouraged.** `gecko_app` is one flat ~162-entry source list; the `vault` I/O library is leaky in both directions (`MapWriter.cpp:8` reaches up into `editor/`; `MapObject.cpp` is forced into `gecko_app`). 123 files use `../../` includes despite both targets publishing their src root. This is PLAN.md item #1 and the gate for fast headless tests.

5. **Establish round-trip and core-logic test coverage that the architecture currently makes impossible.** The most fidelity-critical code (MAP/PRO read→write→read) has no automated test because `MapObject` behavior exiles it into `gecko_app`, and `SelectionManager`/resource cache/FID-resolution are untestable without Qt. The DI and build-split work directly enables PLAN.md item #6.

6. **Collapse the format-model and field-order duplication that breeds silent round-trip drift.** PRO type structs are defined twice (`Pro.h` vs `ProDataModels.h`, already diverged on `weaponFlags`); reader and writer hand-enumerate ~80 fields in lockstep with no shared schema (the source of the wall-SID bug). A single source of truth shrinks the surface that must stay aligned with fallout2-ce.

7. **Pay down the cheap consistency debt that erodes the documented conventions.** Two competing spacing-token namespaces, `emit` vs `Q_EMIT` (113 vs 4), inconsistent `onXxx()` slot prefixes, dead CMake/profiler/Signal scaffolding, and stale CLAUDE.md docs. Individually trivial; together they make the codebase read as "conventions optional."

## Addendum: Gaps & Corrections

The roadmap is well-grounded and constraint-clean; the gaps below are about a missing dimension (concurrency), an over-gated test net, and three mislocated/under-scoped fixes. All are verified against the code.

### A. Concurrency is under-represented beyond the single Loader race (HIGH, missing dimension)
`MapLoader` runs `load()` on a `std::thread` (`MapLoader.cpp:253`) and the worker mutates two pieces of unsynchronized shared state: the `ResourceRepository` cache (via `_repository.load<Frm>()` inside `TextureManager::preload`, `TextureManager.cpp:44`) and `TextureManager::_textures` (`std::unordered_map`, no mutex, `TextureManager.h:34`; `try_emplace` at `TextureManager.cpp:54`), while the render thread calls `textures().get()` (`TextureManager.cpp:57`, itself a non-atomic find-then-try_emplace). WP-9 step 1 treats MapLoader as purely a Qt-coupling problem ("replace `QString().arg().toStdString()`") and never addresses the threading contract. Making the loader "headless" without this risks normalizing the data race. Additionally, `sf::Texture` GPU upload (`loadFromFile`/`loadFromImage`) is GL-context-affine; any headless redesign must keep upload on the context-owning thread.
**Action:** Add an explicit thread-safety sub-item (serialize cache inserts vs render reads, or move insertion to the main thread on completion) and sequence it before/with the MapLoader headless refactor.

### B. The round-trip safety net is NOT gated on WP-8 — un-gate and re-sequence (HIGH)
The roadmap repeatedly claims MAP/PRO round-trip tests are "impossible" / "exiled into gecko_app" and makes WP-9 depend on WP-8. But `ProReader/ProWriter/MapReader/MapWriter` are all already in `vault` (`src/CMakeLists.txt:299-336`), `general_tests` already links `vault` + SFML (`tests/CMakeLists.txt:31-37`), and a PRO read test already exists (`test_reading_pro_drug.cpp`). `MapObject` has no Qt dependency. A `read→write→read` byte-compare for PRO can land today with a committed fixture `.pro`.
**Action:** Move a minimal PRO round-trip assertion into WP-1 so the wall SID/flagsExt fix (`ProWriter.cpp:399-400`, a deliberate on-disk byte change) ships proven-correct rather than with the test deferred "when WP-9 lands." Only the `MapObject` behavior split (`ObjectQueries`) genuinely needs WP-8, and that is off the read/write path.

### C. The MapLoader crash fix is mislocated (HIGH, correctness of the fix itself)
Exec-Summary item 1, WP-1 step 3, and the Quick Win cite `MapLoader.cpp:148` as "outside any try/catch." Line 148 is the `MapReader::openFile()` call, which is already inside a try/catch (`MapLoader.cpp:137-152`). The actual unguarded throw escapes from `loadMapResources()` — called *after* that try block on the filesystem path, with its preload loops at `MapLoader.cpp:200,225` unguarded. (`loadFromVFS` already wraps `loadMapResources`, so only the FS path is exposed.) The prescribed fix "wrap `loadFromFilesystem`'s body" would not catch it.
**Action:** Wrap `loadMapResources()` (the shared completion path) or the top-level `load()` dispatch.

### D. "Throw on unknown object type" must target the OUTER switch only (MEDIUM)
`MapReader.cpp:138-152`: the MISC inner-switch `default` deliberately falls through to read four exit-grid fields for object IDs 16–23 *and* unknown subtypes. The throw belongs only in the OUTER `OBJECT_TYPE` default at `:155`. WP-1 step 5's "match sibling switches" phrasing is ambiguous; an over-eager edit to the MISC inner default would break exit-grid parsing.

### E. FrmSelector sentinel is multi-site and collides with real low PIDs (MEDIUM)
`deriveFrmPidFromPath` returns `0` at `FrmSelectorDialog.cpp:515,534,577` (distinct failure causes) but also returns legitimate low PIDs like `0x00000001` (`:539`). The `std::optional` migration must distinguish "derivation failed" from real low-value PIDs; a mechanical `>0` swap risks masking a valid PID as failure.

### F. Metric drift (LOW, confidence)
Re-verified counts differ from the roadmap: `emit` ≈107 vs `Q_EMIT` ≈2 in src (roadmap: "113 vs 4"); `../../` relative includes in 121 files (roadmap: 123, cited twice); `Settings::getInstance` 18 occurrences across 6 files (roadmap: "17 sites"). The qualitative conclusions hold, but the figures were not re-validated. Separately, `qt_tests` is a **live** target (`tests/CMakeLists.txt:26-56`, linking `gecko::app` with an offscreen Qt harness) — the roadmap inherits CLAUDE.md's stale "no qt_tests target" claim and so undersells existing Qt-test infrastructure that could host SelectionManager tests incrementally.
